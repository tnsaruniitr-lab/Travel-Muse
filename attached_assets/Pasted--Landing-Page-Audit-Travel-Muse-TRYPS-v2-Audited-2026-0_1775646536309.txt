| `section-break` | 2.0MB | **249KB** | 88% |
| `hero-bg` | 1.6MB | **116KB** | 93% |
| `how-to-plan-group-trip-hero` | 1.5MB | **96KB** | 94% |
| `group-trip-expense-tracker` | 1.5MB | **90KB** | 94% |
| `group-trip-date-poll` | 1.2MB | **45KB** | 96% |
| `shared-group-trip-itinerary` | 1.1MB | **51KB** | 95% |
| `app-mockup` | 1.1MB | **43KB** | 96% |

**Total blog image payload: ~1.7MB (down from 26MB). Blog LCP on mobile now viable.**

**Problem:** The original `.png` files are **still in `public/images/`** alongside the WebPs. They serve no purpose — the `.tsx` pages reference `.webp` exclusively. They add 26MB of dead weight to the repo and risk being served accidentally. Delete them all.

`oahu-north-shore-day.webp` at 779KB is still the heaviest image by far — it's more than all other images combined. A re-compress at quality 75 should get it under 200KB with minimal visible difference.

### 3.2 Security Headers — Partially Resolved

`helmet` is installed and active. This adds:
- ✅ `X-Frame-Options`
- ✅ `X-Content-Type-Options`
- ✅ `Strict-Transport-Security`
- ✅ `Referrer-Policy`
- ✅ `X-DNS-Prefetch-Control`

**But CSP is disabled:**
```ts
app.use(helmet({
  contentSecurityPolicy: false, // ← XSS protection off
}));
```

This is understandable — CSP is hard to configure correctly with a React app that uses inline styles, Google Fonts, etc. But disabling it entirely removes the most important XSS protection helmet provides.

**Recommended approach — start with report-only mode:**
```ts
app.use(helmet({
  contentSecurityPolicy: {
    directives: {
      defaultSrc: ["'self'"],
      scriptSrc: ["'self'", "'unsafe-inline'"], // tighten later
      styleSrc: ["'self'", "'unsafe-inline'", "https://fonts.googleapis.com"],
      fontSrc: ["'self'", "https://fonts.gstatic.com"],
      imgSrc: ["'self'", "data:", "https:"],
      connectSrc: ["'self'"],
    },
    reportOnly: true, // log violations without blocking
  },
}));
```
Start in `reportOnly` mode, observe violations in the console, then enable enforcement once clean.

### 3.3 Schema Issues — Multiple Broken Image URLs

Several JSON-LD schema entries reference image URLs that don't exist in `public/`:

**In `home-head.ts`:**
- `"logo": "https://jointryps.com/assets/logo.png"` — no `public/assets/` directory exists
- `"image": "https://jointryps.com/images/tryps-preview.png"` — file doesn't exist
- `screenshot` array: `tryps-date-poll.png`, `tryps-itinerary.png`, `tryps-expense-split.png` — none exist

**In `oahu-group-trip-itinerary-head.ts` and `how-to-plan-group-trip-head.ts`:**
- `"logo": { "url": "https://jointryps.com/logo.png" }` — file doesn't exist

Google's Rich Results Test will flag these. Broken image references in schema don't prevent indexing but they reduce schema quality signals and can cause rich result eligibility issues.

**Fixes:**
1. For the logo: either add a `public/assets/logo.png` file, or update to `/favicon.svg` (which exists)
2. For screenshots: use the actual WebP blog images that exist, e.g. `/images/blog/oahu-group-trip-hero.webp`
3. For blog OG/Twitter images: change all three head files to reference `.webp` instead of `.png`

### 3.4 Blog OG/Social Images Still Broken

The `<img>` tags in blog pages correctly use `.webp`. But the `<meta>` og/twitter tags in the head files still point to `.png`:

```ts
// how-to-plan-group-trip-head.ts
<meta property="og:image" content="https://jointryps.com/images/blog/how-to-plan-group-trip-hero.png" />

// oahu-group-trip-itinerary-head.ts  
<meta property="og:image" content="https://jointryps.com/images/blog/oahu-group-trip-hero.png" />

// index-head.ts
<meta property="og:image" content="https://jointryps.com/images/blog/how-to-plan-group-trip-hero.png" />
```

If the old `.png` files are deleted (which they should be), **all blog social share cards will break**. Even if they're kept, crawlers may be served a 404 from CDN caching. Change these to `.webp` now.

### 3.5 Mobile Nav — Fixed

`home.tsx` now has a `useState(false)` hamburger menu with `aria-controls="mobile-nav"` and `onClick={() => setMenuOpen(o => !o)}`. This resolves the previous gap. Good implementation.

### 3.6 Lazy Loading — Fixed Correctly

`ArticleImage` component defaults to `loading="lazy"` and only the hero `<img>` overrides with `loading="eager"`. This is exactly the right pattern.

### 3.7 Caching — Fixed

```ts
express.static(path.resolve(__dirname, "dist/public"), {
  index: false,
  maxAge: "1y",
  immutable: true,
})
```

This is correct for production — assets get aggressive 1-year caching. The `immutable` flag tells browsers not to revalidate. Vite adds content hashes to bundled filenames, so cache-busting is automatic.

### 3.8 Routing Duplication — Still Present (Low Priority)

The `entry-server.tsx` / `main.tsx` dual route map pattern remains. No new pages have been added so no new bugs introduced. Still worth resolving as the site grows — a shared route config would prevent the inevitable mismatch when a new blog post is added to one file but not the other.
